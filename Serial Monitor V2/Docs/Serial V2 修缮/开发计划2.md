<a id="-快速改进bug-修复--体验硬伤优先做"></a>
## 🔴 快速改进（Bug 修复 / 体验硬伤，优先做）

<a id="1-绘图期间拖拽滑杆不卡"></a>
### 1. 绘图期间拖拽滑杆不卡

**根因**：Plot 和 Sliders 是互斥的两个面板，靠 `Visibility` 切换。切到滑杆页后 `panelPlot` 变成 `Collapsed`，但 `OnPlotMessage` 没停——数据照存、`InvalidatePlot` 照渲。OxyPlot 30Hz 刷新和滑杆 MouseMove 抢同一条 UI 线程，拖拽一顿一顿。

**不卡的条件**：不发绘图数据（波形没跑）、不打开串口（啥都没跑）、只发 slider 数据（波形没跑）——三者共同点：OxyPlot 不在渲染。切到滑杆页后 OxyPlot 看不见但仍在渲染，这是被忽略的第四种"不该卡但卡了"的场景。

**主方案：按面板可见性跳过渲染（推荐）**

`PlotViewModel` 加 `IsActive`，`OnPlotMessage` 里数据路径照跑，只在最后跳过不可见时的 `InvalidatePlot`。

```csharp
// PlotViewModel.cs
public bool IsActive { get; set; } = true;

// OnPlotMessage 末尾
if (IsActive)
    Model.InvalidatePlot(true);
```

```csharp
// MainWindow.xaml.cs 标签切换处
_plotVM.IsActive = (_currentTab == "Plot");
```

**为什么不会像暂停那样压缩**：暂停时数据停止进入 Series → 旧点和新点之间有巨大时间 gap → 窗口被拉伸。离开 Plot 页面时数据没停——`OnPlotMessage` 照跑、Series 照存、sweep 照清，只跳过渲染。切回来没有时间 gap，下一帧自然触发最新 200 点渲染。

**方案组合**：

| 方案 | 适用场景 | 思路 | 代码量 |
|---|---|---|---|
| **IsActive 跳过渲染**（主） | 切到任意非 Plot 页面 | 不可见就不渲，数据照存 | ~5 行 |
| **调度法**（#8 联动） | 调参工作台拖拽滑杆时波形同屏可见 | `InvalidatePlot` → `Dispatcher.BeginInvoke(Background)`，渲染让路给 MouseMove | ~50 行 |

> IsActive 覆盖切页场景：离开 Plot 页面 → 跳过渲染 → 零开销。**不做 #8 的话，IsActive 5 行就封堵了全部拖拽卡顿场景。**
>
> 但做完 #8（调参工作台）后，滑杆和波形**同屏可见**——IsActive 救不了。此时需要调度法：30Hz 触发不变，但 `InvalidatePlot` 用 `DispatcherPriority.Background` 派发，WPF 优先处理 MouseMove，空闲才渲染。波形帧率自然降到 15-25Hz（取决于 UI 线程忙闲），松手瞬间恢复满 30Hz。MouseMove 永远流畅，Series 数据不丢。

**为什么不是降频法**：5Hz 对调参场景太卡——拖滑杆时用户眼睛盯着波形看趋势，5 帧根本不够。降频本质是"牺牲帧率保手感"，调度法是"不牺牲帧率，让系统自己挤时间"。

| | 降频法 5Hz | 调度法 Background |
|------|:--:|:--:|
| 滑杆手感 | 流畅 | 流畅 |
| 波形帧率 | 强制 5Hz | 自然 15-25Hz |
| 松手恢复 | 等下一次定时器 | 立即 |
| 数据丢了吗 | 没丢（Series 照存） | 没丢 |

**代码量**：~55 行（IsActive 5 行 + 调度法 ~50 行） | **难度**：小 | **文件**：`PlotViewModel.cs` + `App/MainWindow.xaml.cs`

<a id="2-发送区-hex-非法字符实时提醒"></a>
### 2. 发送区 HEX 非法字符实时提醒

**现状**：`ValidateHexString` 已存在（`DataConverter.cs:144`），发送时已调用（`MainWindow.xaml.cs:1285`），但结果丢进了系统日志——`LogSystem($"HEX 输入含无效字符，已忽略：{invalidChars}")`。用户盯着发送按钮，不会看日志。

**方案**：系统日志那行不动，额外在发送按钮旁加一个黄色 TextBlock，实时显示。已有 `ValidateHexString` 返回值直接用。

```
"AA BB GG" → [发送]  ⚠️ 无效 HEX 字符: G
```

**代码量**：~10 行（后端已完成） | **难度**：极小 | **文件**：`MainWindow.xaml` + `App/MainWindow.xaml.cs`

<a id="4-版本号自动注入--启动检查更新"></a>
### 4. 版本号自动注入 + 启动检查更新

**版本号自动注入**：csproj 加 `<Version>2.1.0</Version>`（后续发版时改），关于页从 `Assembly.GetEntryAssembly()` 读取，不再手写。

**启动检查更新（依赖版本号自动注入）**：

启动时向 GitHub API 发一次 GET：

```
GET https://api.github.com/repos/Encaron/SerialMonitor/releases/latest
```

拿到的 `tag_name`（如 `v2.2.0`）和本地版本比。如果远端更新，弹窗：

```
┌─ 发现新版本 ───────────────────────┐
│                                     │
│  当前版本：v2.1.0                    │
│  最新版本：v2.2.0                    │
│                                     │
│  [查看更新内容]   [稍后提醒]         │
└─────────────────────────────────────┘
```

点"查看更新内容"→ `Process.Start("https://github.com/Encaron/SerialMonitor/releases/latest")` → 浏览器打开下载页。不做静默下载/自动替换 exe——那要处理 UAC 提权、文件锁、回滚，对项目投入产出不成比例。

**HTTP 请求**用 `HttpClient`（.NET 8 内置），异步发起、不阻塞启动。网络不通/GitHub 挂了/超时 5s → 静默跳过，不影响正常使用。

⚠️ **挨打风险**：GitHub API 对未认证请求限速 60 次/小时/IP。这里每次启动发一次，正常用够。但如果调试期间频繁重启 exe（60 次+/h）→ API 返回 403 → 当作"检查失败"静默处理，不做重试。

**代码量**：~20 行（自动注入 5 行 + 检查更新 15 行） | **难度**：小 | **文件**：`SerialMonitor.csproj` + `App/MainWindow.xaml.cs`

<a id="5-滑杆摇杆侧栏内容充实--摇杆反馈一致性修复"></a>
### 5. 滑杆/摇杆侧栏内容充实 + 摇杆反馈一致性修复

> 合并原 #7b（摇杆侧面板反馈一致性修复）和原 #14（滑杆/摇杆侧栏内容充实）。两项管的是同一块侧栏——先修行为，再改内容。

**摇杆部分（#7b 行为修复 + 视觉改为协议反馈）**：

现状：`RefreshJoystickSideValues()` 在 `MouseMove` 每像素调用，串口没开侧栏也在跳——跟滑杆/按键的"只在发送后更新"不一致。

```
改前：MouseMove → RefreshJoystickSideValues() → 侧栏每像素跳
改后：MouseMove → 大圆盘数字跳（不变）
     发送后  → RefreshJoystickSideValues() → 侧栏更新（跟按键/滑杆一致）
```

同时把侧栏的"实时数值"卡片替换为协议消息反馈卡片，格式跟滑杆反馈统一：

```
┌─ 摇杆反馈 ────────────┐
│ J1 [joystick,1,       │
│    0.35,-0.42]         │
│ J2 —                  │
└───────────────────────┘
```

只在串口打开且发送后刷新。大圆盘底下实时数字保留不受影响。

**滑杆部分（侧栏内容扩写）**：

正常模式下原来只有一个反馈卡片 + 一句操作提示，200px 侧栏用了不到 80px，大量空白。改为：

```
┌─ 快速预设 ──────────┐
│ [全部归零] [全部置中] [全部最大] │
├──────────────────────┤
│ 滑杆反馈              │  ← 已有，保留
│ 最近操作：kp          │
│ 当前值： 27.50        │  ← 新增：拖拽时实时跳
│ 范围：  0 ~ 100       │  ← 新增：选中滑杆的 min/max
│ 步长：  0.5           │  ← 新增
│ 发送内容：[slider,    │  ← 已有
│ kp,27.5]              │
└──────────────────────┘
```

拖哪个滑杆显示哪个的详情。快速预设三个按钮一键批量设值（归零/置中/最大）。⚠️ 拖拽过程中更新活跃详情不能调 `RefreshSlidersUI()`（会销毁正在拖的控件），直接更新 TextBlock.Text。

**行为明确**：
- **初始状态**：侧栏打开时取 `_sliderVM.Sliders[0]` 显示详情（列表非空）。一个 `_lastActiveSlider` 字段，初始 null → fallback 第一个滑杆；用户拖拽后切到该滑杆。
- **"最近操作"** = 最后一次用户拖拽——不是 MCU 回设。MCU 发的 `[slider,kp,50]` 静默更新滑杆位置和值，不算"操作"。
- **快速预设后**：显示第一个被预设的滑杆的详情。

**代码量**：~45 行 | **难度**：小 | **文件**：`MainWindow.xaml` + `Panels/Sliders.cs` + `Panels/Joystick.cs`

⚠️ **侧栏宽度自适应**：侧栏可拖拽调整宽度（GridSplitter）。侧栏变窄时，协议消息文本自然断行——如 `[slider,kp,27.5]` → `[slider,`（换行）`kp,27.5]`。**这不必修**。当前已有的 `TextWrapping="Wrap"` + 等宽字体（`Sarasa Mono SC`）即解决方案。新 AI 不要看到文本断行就以为是 Bug。

<a id="6-oled-绘图指令"></a>
### 6. OLED 绘图指令

**需求**：STM32 通过串口发送绘图原语，PC 端 OLED 面板渲染。支持画点、画线、画圆、矩形、色块填充，以及小尺寸图片上传（≤64×64）。

**协议格式**：
```
[draw,point,  x,y,#RRGGBB]
[draw,line,   x1,y1,x2,y2,#RRGGBB]
[draw,rect,   x,y,w,h,#RRGGBB]
[draw,fill,   x,y,w,h,#RRGGBB]
[draw,circle, cx,cy,r,#RRGGBB]
[draw,clear]
```

**PC 端改动**：
- `MainWindow.xaml.cs`：路由加 `else if (msg.Type == "draw")`
- `Panels/Display.cs`：6 个 handler 函数，用 WPF Canvas Shape 渲染
- 协议解析器零改动（Type 字符串不校验）

**STM32 端封装**（见 `Docs/MCU/OLED绘图协议.md`）：
```c
void Draw_Line(int x1, int y1, int x2, int y2, const char *color) {
    Log("[draw,line,%d,%d,%d,%d,%s]\r\n", x1, y1, x2, y2, color);
}
// 其余 5 个同理
```

**代码量**：~150 行 | **难度**：中 | **文件**：`MainWindow.xaml.cs` + `Panels/Display.cs`

<a id="7--pc-鼠标画板--stm32-物理屏依赖-6"></a>
### 7. ✏ PC 鼠标画板 → STM32 物理屏（依赖 #6）

> ⚠️ 依赖 #6 的 OLED 面板和 `[draw,...]` 协议。用户可能不会立即做——做完 #6 后再展开。当前条目记录完整设计，实现时按此执行。

**需求**：PC 端 OLED 面板加绘图工具栏，鼠标绘制 → 编码为 `[draw,...]` → 串口发送 → STM32 LCD 驱动渲染到物理屏。不用重编译固件即可调 UI 布局，调完导出 C 数组写入代码。

**工具栏**：

```
✏ 画笔  📏 直线  ▭ 矩形  ◯ 圆  T 文字  线宽[1][2][3][5][8]  ■颜色  🧹橡皮擦  [导出C数组]
```

- 颜色复用现有调色盘 UI（按键/滑杆已有 `ShowColorPickerPopup`，~0 行新代码）
- 线宽对全部图形工具生效。屏幕分辨率 ≥ 240×240 时工具栏显示；≤ 128×64 时默认 1px 不暴露
- 橡皮擦 = 画笔颜色设成底色。单色屏擦 = 灭，彩色屏擦 = `#000000`

**两种绘制模式**：

| 模式 | 工具 | 行为 | 发送时机 |
|------|------|------|---------|
| 点击放置 | 📏 直线 / ▭ 矩形 / ◯ 圆 / T 文字 | 点起点 → 拖拽 → PC 本地预览虚线 → 松手定型 | 松手发一条 |
| 自由绘制 | ✏ 画笔 | 按住拖 → 每 ~30ms 采样当前位置 → 连成短线段 | 拖的过程中持续发 |

```
画笔 MouseMove 连续采样：
  (10,20)→(12,22)→(15,21)→... → 每段编码为 [draw,line,旧x,旧y,新x,新y,#RRGGBB]
  手抖了 2px，物理屏上就歪 2px —— 一比一复刻
```

**预览虚线走本地渲染**：点击放置模式下，拖拽时的虚线预览由 WPF Canvas 本地绘制，不走协议、不经过串口。只在松手定型时编码一条 `[draw,...]` 发出。这避免了 MouseMove 每像素都发串口数据的带宽浪费——和自由绘制模式的持续发送是两种不同的数据路径。

**放置后可调整**：直线/矩形/圆放置后不焊死——点一下选中（出现拖拽控点），拖完松手重新发送。PC 画布上的预览只是大概，物理屏才是最终效果。调 UI 布局的正常流程是 PC 上粗拖 → 看物理屏 → 不对 → 拖一下 → 再看 → 对，记下坐标。

**文字输入**：点位置 → 弹出输入框 → 输入文字 + 选字号 → 编码为 `[draw,text,x,y,"text",fontSize,#RRGGBB]`。不在 PC 端选字体风格——物理屏用取模工具生成固定点阵字，PC 端选"宋体/黑体"没意义。

⚠️ **待改项——文字渲染依赖 MCU 字库**：PC 端发出 `[draw,text,10,20,"Hello",12,#FFFFFF]` 后，MCU 端需要 ASCII 字库或取模数据才能渲染。这不是 PC 端能解决的问题——实现时需在 `Docs/MCU/OLED绘图协议.md` 里说明：文本渲染要求固件端集成字库（ASCII 5×7 或 8×16 点阵常见库即可覆盖英文+数字）。

**颜色处理**：PC 端保留颜色选择器。STM32 端按屏的能力处理——单色 OLED 忽略颜色字段照画；彩色屏（ST7789/ILI9341）解析 RGB565 后着色。

**导出 C 数组**：整个功能的终点。`[导出C数组]` 按钮 → 把当前画布内容编码为：

```c
const uint8_t logo[] = {0x00,0xFF,0x81,...};  // 单色屏
const uint16_t logo[] = {0xF800,0x07E0,...};  // 彩色屏 RGB565
```

导出含尺寸注释，直接粘贴进固件。

⚠️ **待改项——导出 C 数组的细节**：导出是把当前画布所有图形**光栅化**为一张位图，再编码为 C 数组——不是逐条 draw 指令的 C 翻译。光栅化用 WPF `RenderTargetBitmap`。实现前需明确：
- Canvas 背景色——导出时包含背景（白色/黑色）还是透明？
- 单色屏导出时阈值——灰度 → 黑/白的判断标准（如 >128 算亮）
- 彩色屏 RGB565 字节序——大端还是小端？

**不做**：
- 字体风格选择（宋体/黑体——物理屏不认）
- 回退/撤销（物理屏已画上去，反向操作复杂，砍）
- 导出 PNG 图片（嵌入式要的是数组不是图像文件）
- Screen View 层级面板（128×64 屏元素不超过 10 个，删了重画更快）

**串口带宽**：115200 波特率 ~11KB/s，画笔采样 60 个点/秒 × 每点 ~40 字节 = 2.4KB，约 0.2 秒发完。921600 波特率下 26ms 发完——带宽不是瓶颈。

**固件端**（~50 行 C）：`Serial.c` 里 `OpMode` 和 `HAL_UARTEx_RxEventCallback`（`SERIAL_REALTIME` 模式，空闲中断收大数据）已存在——这是用户为 `[draw,...]` 预留的接收接口。未来 AI 不要删 `OpMode` 参数。

<a id="8--调参工作台原数据仪表盘"></a>
### 8. 📊 调参工作台（原"数据仪表盘"）

**为什么需要它**：Plot 和 Sliders 是互斥标签页——拖滑杆看不到波形，看波形拖不了滑杆。没有物理电位器的用户调 PID，唯一的路径就是：切滑杆 → 拖 → 切波形 → 看 → 切滑杆 → 再拖……来回切屏。加一个能同时拖滑杆并看波形的页面，是为了用软件模拟"手拧电位器 + 眼看示波器"的硬件调参体验。

**不是**：把四个面板的数据一锅烩的截图看板。**是**：拖滑杆时立刻看到波形变化的调参闭环。

---

**内容选择：哪些滑杆该出现在这里？**

不能把所有滑杆堆进来——`led_pwm` 控制呼吸灯，和波形无关。区分机制用**命名复用**：

```
PC 端滑杆叫 "My_Kp" → [slider,My_Kp,27.5] → STM32 固件
                                           kp = 27.5
                                           printf("[plot,My_Kp,%f]", result)
PC 收到 [plot,My_Kp,0.523] → 画曲线
```

——"My_Kp" 同时有曲线和滑杆 → 它们是外观配对，出现在调参区。`led_pwm` 没有 plot 数据 → 不出现。零新协议，STM32 端零改动。判断逻辑在 PC 端：遍历所有 `Model.Series` 和 `_sliderVM.Sliders`，取名字交集。

---

**两种实现方案**：

| | A. 独立标签页 | B. Plot 底部抽屉 |
|---|---|---|
| 位置 | 新加一个图标，独占主内容区 | Plot 底部可伸缩面板 |
| 交互 | 切标签页进入/离开 | 初始全屏绘图 + 底部 ▲ 按钮；点击弹出抽屉；再点或 Esc 收回 → 恢复全屏 |
| 空间 | 全屏，波形和滑杆各占可调比例 | 弹出后：绘图 3/4 + 滑杆 1/4；拖 GridSplitter 最大到各 1/2；不够则滚轮滚动滑杆区 |
| 图标栏 | 多一个标签 | 不占 |
| 未来扩展 | 所有控件塞一个页面 | 每个场景独立：Plot+调参、OLED+命令输入…… |
| 复用度 | 共用 `_plotVM` + `_sliderVM` 实例，非拷贝 | 同左 |

**抽屉内容**：比 Sliders 页面更紧凑——无卡片边框、无编辑按钮，但保留 **+/- 精确微调按钮**（调 PID 时点一下步进 0.1，看曲线反应）。拖拽粗调、按钮精调，两者并行。

**推荐**：先上 B（抽屉），如果空间不够再考虑 A（独立标签页）。

⚠️ **前置依赖——#1 调度法必须同步上**：抽屉弹出后滑杆和波形同屏可见，拖拽时 OxyPlot 渲染和 MouseMove 抢 UI 线程。`IsActive`（#1 主方案）救不了——波形可见时 `IsActive = true`。必须同步实现 #1 的调度法（`InvalidatePlot` → `Dispatcher.BeginInvoke(Background)`），否则调参工作台的拖拽体验就是一顿一顿的。

⚠️ **关键实现约束 1——VM 共享**：调参条里的滑杆和 Sliders 页面必须共用**同一份 `SliderViewModel` 实例**——`_sliderVM.Sliders[n]` 是同一个对象引用，不是拷贝。这样用户在 Sliders 页面设置的步长/颜色/范围会立刻在调参条生效，反之亦然。新 AI 实现时禁止 `new SliderViewModel()`——直接拿 MainWindow 已有的 `_sliderVM`。

⚠️ **关键实现约束 2——波形压扁不是 Bug**：抽屉弹出后绘图区像素高度变小，同一份数据画在更矮的画布上——曲线视觉上变扁，但 Y 轴范围（`_yAxis.Zoom`）不变，数据 Series 不变，暂停后双击查看详情/CSV 导出/HUD 数值全部不受影响。新 AI 不要看到"波形扁了"就回去调 OxyPlot 参数——这是物理约束，不是 Bug。

⚠️ **关键实现约束 3——抽屉弹出与 #1 IsActive 不冲突**：抽屉弹出/收回时 `PlotView` 控件高度变化，OxyPlot 内部自动重绘以适应新尺寸。此时用户必然在 Plot 页面（`IsActive = true`），不会被 #1 的跳过渲染逻辑拦截。两条机制正交，互不干扰。

⚠️ **关键实现约束 4——+/- 微调按钮的步进值来源**：步进值直接取自滑杆的 `TickFrequency` 属性（`_sliderVM.Sliders[n].TickFrequency`），不新发明参数。如果 MCU 端 PID 是浮点（如 `kp=0.523`），用户可在 Sliders 页面把 TickFrequency 设为 0.001，微调按钮就每次步进 0.001。复用已有属性，零新代码。

**代码量**：~150 行（大量 XAML 排版，少量 C# 胶水） | **难度**：中 | **文件**：`MainWindow.xaml` + `PlotViewModel.cs` + `Panels/Sliders.cs`

### 9. 🔬 协议调试器 → 已砍

**原方案**：侧面板新增页，DataGrid 实时滚动最近 20 条解析后协议消息，类型标色。

**砍掉原因**：各模块侧面板已有独立协议反馈——滑杆有"滑杆反馈"（最近操作+发送内容），按键有"按键反馈"（按下+发送内容）。这些已经覆盖了"每个模块发了什么协议消息"的调试需求。加一个集中式 DataGrid 是把同一份信息讲第三遍（接收区原始文本是第一遍，模块侧栏反馈是第二遍）。不做新页面。

**替代方案——接收区协议高亮 → 也砍了**。用户定调：PC 端发送消息全线蓝色，这是项目设计基调（白+蓝 / 黑+蓝）。多色高亮破坏统一的视觉语言。协议类型区分靠各模块侧栏反馈就够了。

<a id="10--音频频谱页fft-频域分析"></a>
### 10. 📶 音频频谱页（FFT 频域分析）

**需求**：STM32 ADC 采样 → CMSIS-DSP FFT → `[fft,点数,bin0,bin1,...]` → PC OxyPlot 频谱瀑布图。和你现有的 `[plot,...]` 时域波形互补——同一个信号，两种角度看。

**协议格式**：

```
[fft,512,0.12,0.35,0.67,0.89,0.45,...]
       ↑   ↑    ↑    ↑    ↑    ↑
     点数 bin0 bin1 bin2 bin3 bin4 ...（从低到高各频率的幅度，归一化 0~1）
```

**典型调试场景**：
- 查谐波失真——发 1kHz 正弦，FFT 一看有没有 2kHz/3kHz 峰
- 查 ADC 混叠——采样率不够，频谱上不该出现的峰暴露问题
- 查电源噪声——50Hz 工频干扰、开关电源纹波，时域难辨，频谱一个尖峰立刻定位
- 传感器频率响应——扫频看哪个频段最敏感

**PC 端改动**：
- `MainWindow.xaml.cs`：路由加 `else if (msg.Type == "fft")`，数据入 `_plotVM.FreqModel`
- 两个独立 `PlotModel`（`TimeModel` + `FreqModel`），共享一个 `PlotView`。切换时 `plotView.Model` 换引用——因为 X 轴物理意义不同（时间 vs 频率），共用 Model 不成立
- 切换后各自保留缩放/滚动状态

**⚡ 切换按钮设计（经讨论确认）**：

```
┌──────────────────────────────────────────────────────────┐
│ [⏱ 时域 ▾]  │  ⏸ 暂停   🗑 清除   📥 CSV   🔍 适应   📊 详细 │
│  新风格       │         现有按钮不变，统一右移                   │
└──────────────────────────────────────────────────────────┘
```

- **不放在图标栏**——时域和频域是同一信号的两个视角，不是两个功能面板
- **不放在动作按钮中**——切换是"换视角"不是"执行操作"，和暂停/清除不属于同一类
- **位置**：工具栏最左侧，用分隔竖线和动作按钮隔开
- **样式**：新建一个 `PerspectiveToggleStyle`——不是 Primary（那是确认动作），不是 Secondary（那是次要操作）。视觉上介于标签页 tab 和按钮之间——比灰底醒目，比蓝底轻
- **行为**：Toggle 按钮。当前时域显示 `⏱ 时域 ▾`，切到频域显示 `📶 频域 ▾`。点一下切换，PlotView.Model 换引用

**暂停时切换**：两条数据流（`[plot,...]` 和 `[fft,...]`）同时接收、同时存储。暂停冻结两条流的接收，切换只看——切到频域看到的是冻结时刻的频谱快照，和切回时域看到冻结时刻的波形一致。

**📊 详细面板——频域分析指标**：

同一个左侧 `rightPlotDetail` 面板，切到频域后内容换为频域指标：

```
频域（新增）                  vs    时域（现有，不变）
─────────────                       ──────────────
📶 频域                             📊 幅值
  基频：1000 Hz                       Vpp / Vmax / Vmin
  基频幅度：0.92                      Avg / RMS
  THD：0.37%
  DC 偏置：0.01                     ⏱ 时间/频率
  信噪比 SNR：48 dB                   频率 / 周期 / 占空比
                                      Th / Tl / Tr / Tf
📊 数据
  FFT 点数：512                     📊 数据（不变）
  频率分辨率：4.88 Hz/bin             点数 / 范围 / 采样率
  有效带宽：0 ~ 500 Hz
```

- THD = 谐波总功率 / 基频功率，越小信号越纯
- SNR = 基频功率 / 底噪功率，越大信号越清晰
- DC 偏置 = bin0，反映信号的平均直流电平
- 基频 = 有效带宽内最强峰的频率（不是简单的最大值——要排除 DC 和底噪）

⚠️ **架构边界**：所有频域分析指标（基频/THD/SNR/底噪/DC）由 PC 端从 `[fft,...]` 原始数组计算。STM32 只负责采样 → FFT → 发数组，不参与指标计算。

⚠️ **待改项——FFT 帧率物理上限**：
```
512 点 FFT，每点 "0.123," ≈ 6 字符
总消息: "[fft,512," + 512×6 + "]" ≈ 3080 字符 ≈ 3KB
115200 bps / 10 bits per byte ≈ 11.5 KB/s
→ 每秒最多 3.8 帧 FFT
```
3.8 FPS 的频谱更新对调参足够（看频率成分的变化趋势，不是实时动画）。如果未来想做瀑布图（spectrogram）连续滚动显示，需要更高波特率（921600）或减少 FFT 点数（256 点 ≈ 7.5 FPS）。当前默认不做瀑布图，此上限不构成瓶颈。

⚠️ **待改项——基频检测算法细化**：文档定义"基频 = 有效带宽内最强峰的频率（排除 DC 和底噪）"。实现时建议：
- DC = bin 0，直接排除
- 底噪阈值 = 所有 bin（不含 DC）的中位数 × 2——高于此值的峰才算信号
- 如果用户反馈阈值不准，再改为可配置参数

**交互模式——四个面板状态对照**：

切换按钮（`⏱ 时域 ▾` / `📶 频域 ▾`）决定 `rightPlot` 和 `rightPlotDetail` 各显示哪套内容。结构复用，内容切换——和按键面板编辑/正常模式同一种机制。


**砍掉原因**：侧面板背后是空的 Grid 列——没有内容可透。真正 Acrylic 需 `AllowsTransparency="True"`，这是 WPF 性能大坑（拖拽卡、最大化动画坏、WindowChrome 冲突）。侧面板已有 `CardBorderStyle` 圆角卡片，和 VS Code 风格一致，无需再动。

### 14. 窗口圆角 → 已砍

**原需求**：`WindowChrome` + `CornerRadius="8"`，现代 Windows 11 风格。

**砍掉原因**：Windows 11 DWM 已自动给窗口加圆角——用户确认实际运行就是圆的。无需自绘标题栏折腾 `AllowsTransparency`。

<a id="15-面板路由从-mainwindow-抽出"></a>
### 15. 面板路由从 MainWindow 抽出

**方案（经讨论确认）**：采用 **方案 A——搬函数体，路由 switch 保留**。

- 各 handler（`OnPlotMessage`/`HandleKeyMessage`/`HandleSliderMessage` 等）的**函数体**移入对应 ViewModel
- MainWindow 保留一个 `RouteMessage(type, args)` 方法，内部用 switch 派发
- `RouteMessage` 作为活接头：未来想升级为注册表（方案 B），只改这一个方法内部实现
- 当前 7 种 type，最多 ~12 种——switch 够用

**为什么不现在做注册表**：注册表（Dictionary 自动派发）更好扩展，但初始化顺序、冲突处理、handler 签名约束等边界情况对新 AI 容易出错。当前消息类型不多，switch 更直白，debug 更快。

⚠️ **待改项——搬函数体前先 grep 依赖**：各 handler 内部可能引用 MainWindow 控件（`Dispatcher.Invoke`、`this.xxx` 控件名）。搬之前先对每个 handler grep `this\.` 和 `Dispatcher`，列出所有 MainWindow 依赖，再决定是通过参数传入还是接口注入。这是 300 行迁移中最容易出错的部分。

**代码量**：~300 行 | **难度**：中 | **文件**：`App/MainWindow.xaml.cs` + 各 ViewModel

<a id="16-主题切换刷新优化"></a>
### 16. 主题切换刷新优化

**原计划**：`RefreshKeysUI()` 全量重建 → `INotifyPropertyChanged` + DataTemplate 绑定。消除切主题闪烁。

**经讨论改为方案 B——遍历换色，不重建结构。**

**方案 B 做法**：

切主题时不再调 `RefreshKeysUI()` / `RefreshSlidersUI()` / `RefreshJoystickUI()` / `RefreshOLEDUI()` 全量 Clear+重建。改为遍历已创建控件，逐个更新其 Brush 引用：

```csharp
// ApplyTheme 末尾，替换 4 个 Refresh*UI() 调用：
foreach (var btn in _keyButtonMap.Keys)
    UpdateButtonTheme(btn);          // 重取 Background/Foreground/BorderBrush
foreach (var slider in _sliderControlMap.Values)
    UpdateSliderTheme(slider);       // 同上
foreach (var joy in _joystickControlMap.Values)
    UpdateJoystickTheme(joy);
foreach (var elem in _oledElements)
    UpdateOLEDElementTheme(elem);
```

需要维护控件→引用的映射表（`_keyButtonMap` 已存在，滑杆/摇杆/OLED 各自建一个 Dictionary），但映射表本身在 Refresh 时已更新，不增加维护负担。

**为什么放弃方案 A（全面绑定化）**：

| 层面 | 说明 |
|------|------|
| **行为正确性** | 面板 2879 行代码（Keys 912 + Joystick 918 + Sliders 771 + Display 181）是 90+ 个 commit 积累下来的，内含大量交互细节——6 种按键布局自动检测、摇杆三种风格实时切换、拖拽中不重建、编辑态二次确认。方案 A 要把这些全部翻译成 XAML DataTemplate + DataTrigger，丢掉的是已跑通的行为正确性 |
| **测试盲区** | 项目 72 条测试覆盖 ViewModel，不覆盖 UI。方案 A 写出来的新模板，测试全绿但实际交互有 bug——只能靠用户双击 exe 逐个场景验收 |
| **投入产出** | 方案 A 实际远不止 400 行（4 个 ItemsControl + 6 种布局 TemplateSelector + ViewModel INotifyPropertyChanged + 拖拽/动画在模板里的行为适配），为消除 200ms 的切主题闪烁投入这些不值得 |
| **历史教训** | Git 历史显示，每个新交互类型（素材系统、调色盘、布局预设）都是增量加上去的。方案 A 的绑定化会让未来加新交互时链路变长（C# if/else → XAML DataTrigger + Converter），改 bug 时定位更难 |

**代码量**：~80 行（4 个 UpdateXxxTheme 方法 + 映射表补充） | **难度**：小 | **文件**：`Views/Theme.cs` + `Panels/Keys.cs` + `Panels/Sliders.cs` + `Panels/Joystick.cs` + `Panels/Display.cs`

<a id="17b-设置页使用示例重构为折叠分组"></a>
### 17b. 设置页使用示例重构为折叠分组

**问题**：当前设置页使用示例按协议类型逐条平铺展示。随着新标签页增加——FFT 频谱（`[fft,...]`）、OLED 绘图（`[draw,...]`）、传感面板（`[sensor,...]`/`[ctrl,...]`）——v2.7.0 后将达 9+ 种协议类型，每种若干条示例。用户找特定协议示例需大量滚动，当前平铺布局撑不住。

**方案**：重构为按协议类型分组折叠堆放——默认全折叠，点哪个协议展开哪个。

```
设置页——使用示例（改后）
┌──────────────────────────────────────┐
│ ▶ [plot,...]    波形数据             │  ← 默认折叠
│ ▶ [key,...]     按键事件             │
│ ▶ [slider,...]  滑杆                 │
│ ▼ [sensor,...]  传感数据             │  ← 已展开
│   [sensor,temp,芯片温度,42.5,45.0]   │
│   [sensor,humidity,ambient,68.2]     │
│   [sensor,pressure,大气压,1013.2]    │
│   [sensor,status,主板,online]        │
│   [sensor,motor,电机1,2450,85]       │
│   [sensor,generic,电池电压,3.72]     │
│ ▶ [ctrl,...]     控制指令            │
│ ▶ [fft,...]      频谱数据            │
│ ▶ [draw,...]     绘图指令            │
│ ▶ [joystick,...] 摇杆               │
└──────────────────────────────────────┘
```

- **默认全折叠**——和 VS Code 设置搜索同交互模式：不需要看全部，需要时精准定位
- 展开/折叠用 Expander 或 Click 切换 `Visibility`
- 新增协议类型时只加一个折叠组，不破坏已有布局

**触发时机**：v2.7.0 传感面板做完后此问题才会暴露（sensor 和 ctrl 两种新协议类型 + 8 种子类型示例）。v2.2.0~v2.6.0 期间协议类型增量不大（仅 `[draw,...]` 和 `[fft,...]`），当前平铺布局尚可接受。

**代码量**：~60 行 | **难度**：小 | **文件**：设置页 XAML + CS

### 18. CardViewModel 基类抽象（远期，现在不动手）

**核心思想**：`SensorCardViewModel` 里只有 `GetAccentHex` 和 `GetUnit` 跟传感器有关。其余——Type/Name/Value/History/序列化/DeletedNames——是任何"收到消息、知道长什么样、有独立生命周期"的卡片都需要的。把它们往上提一层 `CardViewModel` 基类，8 种传感卡变成 8 个子类覆写 3~5 个方法，`CreateSensorCard` 的 switch 被多态取代。

**基类只放数据和自知，渲染逻辑不往上提**：

```csharp
public abstract class CardViewModel
{
    // 身份
    public string Type { get; set; }
    public string Name { get; set; }
    public string Value { get; set; } = "--";
    public string AuxText { get; set; }
    // 记忆
    public FixedSizeQueue<double> History { get; } = new(30);
    public DateTime LastSeen { get; set; }
    // 自知——子类必须回答
    public abstract string GetAccentHex(bool isDark);
    public abstract string GetUnit();
    public virtual int CardWidth => 160;
    // 序列化
    public Dictionary<string, object> ToDict() { ... }
    public static CardViewModel FromDict(Dictionary<string, object> d) { ... } // 工厂
}
```

子类例子：

```csharp
public class SliderCard : CardViewModel
{
    public override string GetAccentHex(bool isDark) => isDark ? "#5C6BC0" : "#3F51B5";
    public override string GetUnit() => " RPM";
    public override int CardWidth => 280;
}
```

**为什么现在不动手**：

| 理由 | 说明 |
|------|------|
| 只有一个 Cluster 面板 | 传感面板是唯一用卡片基类的。第二个还没来 |
| 8 张卡的基类推导不可靠 | 用 8 种卡归纳出的属性（HasWaveform/HasProgressBar/HasInteractiveControl）碰到第 9 种大概率要改 |
| 提前抽象 = 猜错的概率 × 回退成本 | 等调参工作台/FFT 卡组真的需要，从已跑通的 `SensorCardViewModel` 往上提 120 行，比现在空猜边界安全 |

**触发时机**：第二个 Cluster 面板（调参工作台独立标签页、FFT 卡组、OLED 图层卡组）中的任意一个开始做时，再提基类。

**这条路不是新方向**："一个标签页 = 一种面板"这个假设已经被推倒两次了——#8 调参工作台（Plot + Sliders 同屏）、#19 传感面板（8 种异质卡片混排）。终点是**自定义仪表盘标签页**——用户自己拼的容器：时域图卡、频域图卡、调参滑杆卡、传感器卡、开关卡混在一个 WrapPanel 里，可删可排。是同一条路上的第三步，不是另一条路。

**唯一的新问题**：仪表盘需要 **OxyPlot 卡**——完整带坐标轴、缩放、HUD 的 PlotView 塞进 400×300 卡片。这远比迷你波形（Canvas Polyline 30 点 140×40）重。基类需要一层渲染策略：迷你 Canvas vs 完整 OxyPlot vs 无图。这是传感面板里不存在的东西，也是仪表盘独有的架构增量。

**不会影响前 7 个 Array 面板**：按键/滑杆/摇杆/OLED/波形/接收区/设置是 Array 思维（同质控件排列），不是 Cluster 思维（异质卡片混排）。CardViewModel 是为 Cluster 设计的，7 个 Array 面板不需要继承它。

<a id="-暂时不做"></a>
## ⏸ 暂时不做

### 多串口

**起源**：这个想法比协议系统更早。做 V2 时用户以为"两个波形需要两个串口"——`Serial_Printf(&huart1,...)` + `Serial_Printf(&huart2,...)` 各自发各自的数据。

**为什么被替代**：后来看到江科大 B 站蓝牙串口视频，意识到一个串口可以通过协议格式发多条数据——`[plot,name1,value1][plot,name2,value2]`。协议系统本身就是多串口问题的答案。

**当前不需要的原因**：单人单板调试场景下，一块板子一个 COM 口完全够用。真需要同时监控两个串口时——**同时开两个 exe 窗口各自连各自的串口**就是最简单的多串口方案。操作系统帮你做了进程隔离，不加一行代码。

**什么时候重开**：关键不是"几块单片机同时发数据"——两个 exe 各管各的永远是最简方案。多串口的**真正门坎**是：**数据需不需要汇合到同一个视图？** 比如两块 MCU 各发 `[plot,ch1,...]`，需要叠加在同一张波形图上对比时序关系——那才需要改代码，因为一个 exe 只能开一个串口，两个串口的数进不到同一个 Plot。如果只是各自看各自——永远不需要。**

**给新 AI**：多串口这个条目是 V2 诞生时用户写入文档的——它在协议系统发明之前，本质是用户早期困惑的化石。不要因为它出现在文档里就当成待做需求。如果未来用户真要它，先问："你需要两边数据汇到同一张图吗？"——答案决定了是多串口还是多窗口。

<a id="-架构考量渲染引擎对比"></a>
## 🔬 架构考量：渲染引擎对比

> 波形面板当前用 OxyPlot 2.2，30Hz 软件渲染。拖拽滑杆/摇杆时偶有顿挫感——但不发绘图数据或只发 slider 数据时不卡。根因是 UI 线程争抢（OxyPlot 的 CPU 渲染短暂阻塞了 MouseMove），不是 OxyPlot 性能瓶颈。但为长远考虑，整理了全部可选项。

### 全量对比

| 库 | 授权 | 渲染 | 实时性能 | 社区实测 | 代码量 | 结论 |
|---|---|---|---|---|---|---|
| **OxyPlot**（当前） | MIT 免费 | CPU GDI+ | 64 点/3s × 1 小时不卡 | "不好用"也有"稳"也有 | 0 | **留下** |
| **调度法** | — | 同 OxyPlot | 不改引擎，改排队 | 解决根因 | ~50 | **先上** |
| **ScottPlot** | MIT 免费 | CPU/GPU | 千万点毫秒渲染，Signal 图专为实时优化 | "API 4 和 5 差距大" | ~750 行迁移 | **锦上添花** |
| **LiveCharts2** | MIT 免费 | GPU（SkiaSharp） | 2 点/秒几分钟就卡，1Hz 12-16% CPU | 多人实锤 + GitHub Issue 佐证 | ~520 行迁移 | **排除** |
| **SkiaSharp** 自建 | MIT 免费 | GPU（SkiaSharp） | 完全控制，无中间商 | 自己踩全部坑 | ~950 行重写 | **自由但孤独** |
| **SciChart** | $1,000+ | GPU DirectX | 专业级 | "掏钱不用多想" | 商业集成 | **太贵** |
| **LightningChart** | $1,500+ | GPU DirectX | "不在一个档次" | 顶级 | 商业集成 | **太贵** |
| **ECharts** | Apache 2.0 | JS/Canvas | C#↔JS 桥损延迟 | 有人推荐嵌入 | WebView2 ~150MB | **杀鸡用牛刀** |

### 各方案一句话

- **调度法（先上）**：卡顿是 UI 线程争抢，不是渲染慢。`DispatcherPriority.Background` 让渲染不抢 MouseMove，手永远流畅，波形用空闲刷新。50 行，不改引擎
- **ScottPlot（后补）**：每帧 1-3ms 比 OxyPlot 快 5-10 倍，但还是在同一条 UI 线程排队——调参拖拽时不用调度法照样微顿。调度法 50 行先上，不够再换。迁移 ~750 行
- **LiveCharts2**：GPU 加速≠实时性能好。数据路径 GC 压力大，你的 100Hz 多通道场景撑不住
- **SkiaSharp 自建**：没有任何图库中间层，性能最好、自由度最高。代价是坐标系/刻度/滚动/HUD/Tracker/CSV 全部手写，出 Bug 只能靠自己
- **ScottPlot vs SkiaSharp**：前者是别人造的轮子有人维护，后者是自己的轮子自己扛。性能差距在你这个数据量下感知不到
- **商业库**：等哪天靠这个项目赚到钱了再说

**结论**：调度法先上 → OxyPlot 留下来。如果调度法之后还觉得波形不够流畅（不是卡顿——卡顿已死），再试 ScottPlot 换引擎。SkiaSharp 自建只作为终极备选。
