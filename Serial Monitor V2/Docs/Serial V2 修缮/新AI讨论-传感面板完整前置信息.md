## 一、协议解析——唯一规则

**逗号是唯一分隔符。** `ProtocolParser` 不推断字段语义，只按逗号切、按位置赋值。

```
[sensor, 子类型, 卡片名, 值, 辅助参数]
[ctrl,   子类型, 卡片名, 动作]
```

### 规则表

| 规则 | 说明 | 示例 |
|:-----|:-----|:-----|
| **字段顺序固定，不可换位** | 5 或 4 个逗号分隔位置 | — |
| **末尾字段可省略** | Args 不够 5 个 → 后面全 null | `[sensor,humidity,ambient,68.2]` → 辅助 = null |
| **中段留空用双逗号** | `,,` → 中间字段 = `""` | `[sensor,temp,温度,,45.0]` → 值 = `""`，辅助 = `"45.0"` |
| **名字不可含逗号** | 含逗号 → 全串位移 | `[sensor,temp,芯,片,42.5]` → 6 字段全错 |
| **小数点不是分隔符** | `.` 不被切割 | `42.5` 是一个完整字段 |

### 中段留空的用途

当某个值传感器没读到，但其他值正常——固件写两个逗号紧挨：

```c
// 正常：两个值都有
Serial_Printf("[sensor,temp,温度,%.1f,%.0f]\r\n", temp_value, duty_pct);
// → [sensor,temp,温度,35.0,45]

// 温度值失效，只发百分比
Serial_Printf("[sensor,temp,温度,,%.0f]\r\n", duty_pct);
// → [sensor,temp,温度,,45]
```

PC 端收到 `value=""` → 卡片主值显示 `--`，不进入 History（不画波形）。`aux="45.0"` → 辅助行照常显示。

**不要发明新分隔符或字段序位移逻辑。逗号规则已定。**

## 二、八张卡片——参数全部填满时显示

共性：高 200px、左竖条 12px 贯穿全高、圆角 8px、竖条色=主值色=波形色（三处同源 `GetAccentHex()`）。亮色白底暗色深灰底，切主题自动跟。单位由 `GetUnit()` 返回，渲染时追加到大数字后。波形 = 面积图（Polygon 填满到 Canvas 底部，线条 1px + 填充 30% 透明度）。

### 2.1 温度 `[sensor,temp,芯片温度,42.5,45.0]` — 黄 `#FFC107`

大数字 42.5°C + 辅参 45.0 + 黄色面积波形。辅助有则显无则 Collapsed。

### 2.2 湿度 `[sensor,humidity,ambient,68.2]` — 蓝 `#2196F3`

三层：大数字 68.2% → 进度条(蓝填充/灰剩余，旁标 68.2% 小字) → 蓝色面积波形。

### 2.3 气压 `[sensor,pressure,大气压,1013.2]` — 青 `#00BCD4`

三层同湿度。百分比 PC 端算（值/1013.25×100），固件只发原始 hPa。

### 2.4 电机 `[sensor,motor,电机1,2450,85]` — 紫 `#9C27B0`

和温度同结构：大数字 2450 RPM + 辅参 PWM 占空比% + 紫色面积波形。

### 2.5 状态 `[sensor,status,主板,online]` / `[sensor,status,主板,alarm,overheat]`

online/ok/running → 绿 `#4CAF50`。alarm/error/offline → 红 `#F44336`。无波形。**offline 由 PC 端超时判（2s 没收到心跳自动标红），固件不发 offline。**

### 2.6 开关 `[ctrl,led,蓝色LED,on]` — ON=橙 `#FF9800` / OFF=灰 `#9E9E9E`

胶囊滑块（iOS 风格），60×30 圆角轨道 + 20×20 圆形滑块。ON=橙底白块右对齐，OFF=灰底灰块左对齐。协议大类 `ctrl`（不是 sensor）。点滑块反转状态 → 串口发 `[ctrl,led,名,off]`。无波形。

### 2.7 滑杆 `[ctrl,slider,电机1目标,2450]` — 靛蓝 `#3F51B5`

**280×200**（第一张覆写宽度的卡）。PC→MCU: `[ctrl,slider,名,值]`，MCU→PC: `[sensor,slider,名,值]`。布局：`[+] 2450 RPM [−]` ± 微调按钮夹数值 → Slider 控件 → min/max 标尺。无波形。Min/Max/Step 编辑模式侧栏详情面板设，默认 0~100/步长 1。第一期独立模式。

### 2.8 通用 `[sensor,generic,电池电压,3.72,85]` — 自选色，默认灰 `#607D8B`

专卡的兜底。非标传感器统一发 `generic`，前两字段固定写死。建卡时四可配：名称、单位（自由文本）、颜色（12 色下拉）、波形（勾选框）。160×200，面积图填到底部(不区分 0 基准线)。

## 三、八张卡片共性速查

| 属性 | 值 |
|:-----|:---|
| **高度** | 固定 200px（常量 `CardHeight`，所有类型不覆写） |
| **宽度** | 默认 160px，滑杆卡覆写为 280px |
| **左竖条** | 宽 12px，贯穿全高，纯色填充 |
| **标题** | 卡片名字段，12pt，`TextPrimaryBrush` |
| **核心数值** | 20pt Bold，色 = 竖条色，后跟单位 |
| **辅助参数** | 9pt `TextMutedBrush`，纯数字无文字标签，有则显无则 Collapsed |
| **迷你波形** | Canvas Polygon 面积图 30 点，线 1px + 填充 30% 透明度，填满到 Canvas 底部。无坐标轴/刻度/图例 |
| **圆角** | 8px |
| **主题** | 亮色白底、暗色深灰底，走 `CardBgBrush` DynamicResource |

## 四、未来新建卡片规范

### 4.1 动哪些文件

| 文件 | 加什么 | 行数 |
|:-----|:-----|:-----|
| `SensorCardViewModel.GetAccentHex()` | switch 加一行：新 Type → 色值 | 1 |
| `SensorCardViewModel.GetUnit()` | switch 加一行：新 Type → 单位字符串 | 1 |
| `Panels/Sensors.cs` 的 `CreateSensorCard()` | 加 else if：渲染逻辑（控件组装） | ~30 |
| 协议路由（`OnLineReceived`） | **不动**——sensor/ctrl 大类已扛所有子类型 | 0 |

如果新卡结构 = 现有某卡（如电机卡=温度卡），CreateSensorCard 零改动，走默认分支。

### 4.2 必须遵守的规范

1. **Type 是 string，不是 enum。** 永远小写英文。prefs.json 存字符串，新 Type 不破旧数据。
2. **高度固定 200px。** 宽度可覆写。防止同一排内高低参差。
3. **左竖条色 = 核心数值色 = 迷你波形线色 = GetAccentHex 返回值。** 三处同源。
4. **卡片名字段唯一性**：FindByName 跨所有组搜索。同名 = 同一张卡。改名后旧名失效。
5. **编辑模式下所有交互必须禁用。** 开关/滑杆不可交互。
6. **代码创建的 Popup 必须设 PlacementTarget。**（约束 #24）
7. **中段留空用双逗号**：`[sensor,temp,温度,,45.0]` → 值="" 显示 `--`，aux 正常。**不要发明新分隔符或字段序位移逻辑。**
8. **协议大类归属**：被动接收 → `sensor`；主动发送 → `ctrl`；双向(滑杆) → 两个都用。
9. **波形 = 面积图**：Polygon 填满到 Canvas 底部，不区分 0 基准线。专卡固定有/无，generic 自选。

### 4.3 五问讨论法——和用户逐项确认

| 顺序 | 问题 | 举例 |
|:-----|:-----|:-----|
| **① 协议格式** | `[sensor,Type,Name,Value,Aux?]` 还是 `[ctrl,Type,Name,Action]`？ | 几个字段？每个什么含义？ |
| **② 参数规则** | 哪些必发？哪些可选？哪个可能中段留空？ | 值必发，百分比可选 |
| **③ 卡片视觉** | 竖条色 → 数值色 → 宽高 → 有波形？→ 特殊控件？ | 紫色，240×200，无波形，Slider 控件 |
| **④ 协议归属** | sensor？ctrl？双向？ | 滑杆双向，温度单向 |
| **⑤ 写进文档** | 设计方案 + 开发计划2，不靠聊天记录 | 更新第四章 + 决策表 |

## 五、当前已定事项（不要推翻重来）

### 5.1 侧栏三态模型

`[编辑]` 按钮仅在主区顶部工具栏（和按键/滑杆标签页统一）。

| 态 | 侧栏内容 | 触发 |
|:--|:---|:---|
| **态1 正常模式** | 卡片概览 + 快速跳转（竖条色圆点 + 名 + 值右对齐） | 默认 |
| **态2 编辑-行管理器** | 组管理 + 卡片 ↑↓ 排序 + `[+卡片]` + 滑杆卡 `[⚙]` 齿轮 | 点 `[编辑]` |
| **态3 编辑-卡片详情** | 仅滑杆/通用卡：Min/Max/Step（滑杆）或名称/单位/颜色/波形（通用） | 态2 点 `[⚙]`，← 返回态2 |

### 5.2 组内换行

`"---"` 换行标记——侧栏 `[+ 换行]` 插入。同一灰底框内另起一行 WrapPanel。滑杆卡 280px 独占一行，三张 160 卡一行刚好。

### 5.3 其余已定项

数据结构（组/排/卡片三层）、编辑模式入口/交互/间距、两种 `[+]` 插卡、F2 改名、窗口缩放行为——全部已定，详见设计方案第七、六章。

## 七、Encaron 工作风格提示（给新 AI 看）

- **嵌入式开发者，不写 C#。** 给清晰选项，不要让他读代码才知道什么意思。
- **对 UI 细节敏感**（颜色、对齐、命名），不会用设计术语——**讲效果不讲原则。**
- **做决定可能反复**——看到方案后需要时间想。想好了会说"可以/同意/锁"。定了写进文档，别改回来问第二遍。
- **喜欢类比**——把他的硬件经验（I²C、面包板、跳线）类比软件设计，会更快理解。
- **前面 6 个面板是 Array 思维**（同类型排列），**传感面板是 Cluster 思维**（异质混搭）——他最满意的架构设计。新功能不要掐死这个灵活性。
- **加新卡走五问讨论法**（第四章 4.3），逐条确认不要跳。
