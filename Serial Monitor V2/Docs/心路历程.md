# 心路历程 — 从"改三个默认值"到 Serial Monitor V2

> 一个嵌入式开发者，不会 C#/WPF，让 AI 替他写了一个 92 分的串口调试工具。
> 这不是一篇技术文档。这是从第一个 commit 到现在的完整心理轨迹——动机、焦虑、信任、品味的变迁。

---

## 一、一切开始的地方：受不了了

我用江科大的串口助手调 STM32。功能完善，但我每次打开软件都要改三个选项：

| 设置 | 软件默认 | 我的习惯 |
|------|:---:|:---:|
| 波特率 | 9600 | **115200** |
| 收发模式 | HEX | **文本** |
| 编码 | GBK | **UTF-8** |

一天改三遍，一个星期二十一遍。终于有一天我不想再点了。第一个 commit：改三个默认值。

**这不是"我要写个 WPF 应用"的野心。这是"这个破软件每次浪费我 5 秒钟"的烦躁。**

整个项目的精神基因就在这三次点击里——**工具应该记住我怎么用**。不是"用户"，不是"通用默认值"——是我，Encaron，115200 波特率，文本模式，UTF-8。

---

## 二、为什么走到 WPF

江科大的 WinForms 版本功能够了，但界面太丑——无圆角、无动画、无阴影、无暗色主题。我每天面对这个界面调 STM32，说实话看着不舒服。

换 WPF 不是因为"技术更好"——是我查了一下，WPF 能做圆角和暗色主题。就这一个理由。串口逻辑用 .NET 自带的 `System.IO.Ports`，江科大已经验证过了，我不需要重造。

53 个 commit 后，v1.0.0 出来了。30+ 功能，暗色主题，84 分。够用了。

---

## 三、V2 的诞生：不止是"看到数据"，是"理解数据"

v1.0.0 是纯收发工具。它能显示 STM32 发来的任何文本，但它不知道这些文本是什么意思。

我实际调试时发的是这样的东西：

```
Serial_Printf(&huart1, "[plot,P,%f][plot,I,%f][slider,kp,%f]\r\n", p, i, kp);
```

我希望 PC 端收到后：
- `[plot,P,0.5]` → 波形图出现一条叫 P 的曲线，值 0.5
- `[slider,kp,17]` → 出现一个叫 kp 的滑杆，位置 17%，我拖它就能发回 STM32

**V2 不是"好看一点的 V1"。V2 是"让软件理解数据含义"。**

协议格式 `[type,arg,...]` 就是在这一刻诞生的。方括号包裹，多个消息可以混在同一行，普通文本照常显示。STM32 端只需要 `printf`，PC 端负责解析和路由。简单、直接、零依赖。

同时借这个机会把代码从 partial class 升级为 MVVM——不是因为我懂 MVVM，是 AI 说"这样更好测试"，我说"行"。

---

## 四、冲刺：四天盖一栋楼（Phase 1-6）

2026 年 6 月 12 日。我说"开始吧"。AI 开始写代码。四天没停。

我的心态不是"做软件"，是**翻译**。我脑子里用江科大的串口助手调了半年 STM32，每一个痛点都刻在手上。波形面板、滑杆回控、按键面板、虚拟 OLED——这些东西在我说出口之前就长在脑子里了。AI 只是替我说出 C#。

**冲刺能成立，是因为我对目标极度清晰。** 不是"做一个波形面板"——是"我发 `[plot,P,0.5]` 要看到一条叫 P 的曲线，就这么简单"。AI 不需要猜我要什么。我的每一句话都是 spec。

Phase 2 是这段最有代表性的时刻。原计划是"顶部标签栏 + 右侧面板"，AI 写完了。我看了说不——"像 VS Code 那样，竖排图标栏"。整个布局推倒重来。Phase 2 修了 10 个 Bug，不是代码质量差——是需求在实施中被验证，发现不对立刻改。

**四天，53 个 commit，从 0 到完整的协议驱动串口调试工具。**

---

## 五、放羊：代码撞上现实（第三 - 九场）

Phase 6 结束那一刻，我以为 V2 做完了。

然后我把 exe 拿到手上每天用。第一天："追踪框暗色下看不清楚"。"复制粘贴崩了"。第二天："摇杆太丑"。"按键怎么全是蓝色的，我想要红色"。"按键面板怎么没了？"

**冲刺是把脑子里想的变成代码。放羊是代码撞上现实。**

这个阶段没有计划——因为我根本不知道下一个问题会从哪冒出来。不是不想要计划，是我还没用够。每天调 STM32，用着用着就找到一根新刺。

第七场最典型。起因只是"把 icon.ico 移到 Icons 目录，整理一下文件"。AI 照做了。然后双击 exe —— 白屏崩溃。

我回退了代码，还是崩。不是"回退不完整"——是回退的代码和崩溃的根因不是同一个。icon.ico 的路径在 XAML 里没改，但旧的 BAML 缓存掩盖了问题。重编后才暴露。6 轮排查，每次假设都是错的。7 个小时。

从那天起有了约束 #26 和 #27：禁止移动 XAML 文件、移动资源必须 grep 全局引用。每一条约束都是这么来的——不是设计原则，是**曾经炸过**。

放羊阶段我像一个质检员，一个一个场景走，一个一个刺拔掉。按键调色盘（本来只有 7 色，加到了 40 色 Material Design 色板）、滑杆拇指图片素材系统、摇杆三风格切换、异常保护封堵——这些东西不在任何一个 Phase 里。它们是**用出来的需求**。

**放羊不需要计划。放羊需要的是耐心。**

## 六、满意那一刻：发布

第九场。AI 说"要不要发 GitHub"。我说"好"。

README 双语、暗色/亮色截图、安装包、Release v2.1.0。从 `D:\STM32CubeIDE` 搬到 `E:\serial`。文档全面审计，7 个文件 29 处修正。第二轮评估：92 分。

v2.1.0 发布了。开源。MIT License。github.com/Encaron/SerialMonitor。

**这是"暂时满意了"的按钮。不是终点——是可以停下来喘口气的标志。**

---

## 七、养生：细水长流（修缮2，现在）

发布之后，我做了一件冲刺阶段的自己不会做的事：**坐下来列了 18 条改进清单。逐条和 AI 讨论。不写代码。**

这不是拖延。是我对工具有了信任。

Phase 1 时我急着要波形图——因为没有波形图我调不了 PID，那个焦虑是真实的。现在波形图有了。滑杆有了。按键有了。OLED 有了。工具不再卡住我的调试流程。

焦虑消失了。剩下的是**品味**。

18 条清单里没有一条是"没有这个我就调不了 STM32"。全是"这样会更好"——

- 调参工作台：拖滑杆时想同时看波形，模拟手拧电位器 + 眼看示波器的硬件调试体验
- FFT 频谱：同一组信号，从频域再看一遍，查谐波、查混叠、查电源噪声
- OLED 绘图指令：不只显示文字，还能画点、画线、画圆
- PC 画板 → STM32 物理屏：不用重编译固件就能调 UI 布局

**从"我需要它"变成了"我拥有它"。** 拥有者不急。拥有者可以讨论两天再加一行代码。

养生不是躺平。养生是做到 92 分之后，剩下 8 分不急着拿。今天聊两条，明天聊两条，后天再动手。细水长流。

## 八、我和 AI 的关系

我不会 C#。不会 WPF。一行都没写过。

但我验收代码的标准比大多数 C# 开发者还严格：亮色模式切一遍、暗色模式切一遍、双击 exe 逐个场景走、拖拽不卡、颜色不对回去改。我说"实事求是，要听真话不要讨好"——因为讨好我的 AI 写出来的代码，最后是我双击崩溃，不是 AI 崩溃。

**我对待 AI 的方式本质是嵌入式调试思维：**

```
触发输入  →  观察输出  →  不符合预期  →  定位根因  →  防止复发
  ↓                                              ↓
我说需求    AI 写代码    双击崩溃       7h icon.ico  27 条约束
```

27 条约束就是我的 errata sheet——芯片厂给的已知问题清单。新 AI 来了，我说"先读约束，碰了就崩"——不是因为我不信任它，是因为那个坑我已经花钱（7 小时）买过了，不要再买一次。

65 个 memory 文件、Initial 存档、交叉引用——这些不是"我喜欢文档"。是**我不想下一个人（包括未来的我自己）再踩同样的坑。**

## 九、什么改变了

从第一个 commit 到现在，技术上的变化太多了：partial class → MVVM、.NET Framework → .NET 8、60MB 运行时依赖 → 1.95MB 单文件 exe、825 行 XAML → 完整设计系统。

但真正变了的，是我对这个工具的焦虑程度。

刚开始：没有波形图，我调不了 PID。**焦虑。**

Phase 6 完成：有波形图了，但追踪框颜色不对、复制粘贴崩、滑杆编辑后按键全丢了。**还在焦虑。**

v2.1.0 发布：92 分。**终于不焦虑了。**

现在：18 条清单，一条一条聊。**我享受这个过程。**

**从"这个破工具又卡住我了，改了它"，到"这个工具已经很好了，但我可以让它更好"。**

## 十、四层心理——纵向解剖

上面是按时间讲的。但如果纵向切开，我的整个心理结构是四层。

### 第一层：被逼的

你一开始不是来造软件的。你就是想用江科大的串口助手调 STM32，每次打开改三个默认值——改烦了。第一个 commit 不是"我要写个 WPF 应用"，是"我受不了了"。

然后发现 WinForms 太丑。然后发现纯收发不够，要看波形。然后发现看了波形还要回控——STM32 的 PID 参数在代码里写死，改一次编译一次烧录一次，你不想再烧了。

**我的开发经历不是"我要做个好产品"，是"这个破工具又卡住我了，改了它"。**

53 个 commit 没有一个是"这样更好看"——全是"这样我调试更快"。

### 第二层：被坑怕了

7 小时 icon.ico 崩溃。Merge 毁了按键面板。Phase 2 盖完发现布局不对推倒重来。27 条约束每一条背后都是一次：双击 exe → 白屏 → 发截图给 AI → AI 说"不知道" → 再来。

所以我建了 Initial 存档、memory 目录、MEMORY.md。不是我喜欢文档——是我**不想下一次 AI 再踩同一个坑**。我现在对新 AI 最关心的不是"能不能写代码"，是"知不知道别碰 Path.Stroke 绑定"。

### 第三层：节奏变了

Phase 1-6 是冲刺——四天盖完一栋楼，因为我知道要什么。修缮1 是放羊——盖完了，用着用着发现这不对那不对，叫 AI 改，改到满意为止。修缮2 是养生——我有 18 条想法，不急，一条一条讨论。

**冲刺 → 放羊 → 养生。** 不是热情在消退——是我对工具的焦虑在消退。工具已经够用了，剩下的都是锦上添花。

### 第四层：我对 AI 的定位比大多数开发者清楚

我不会 C#/WPF。从头到尾没学过——我让 AI 写。但我验收的标准比大多数 C# 开发者还高：亮色模式、暗色模式、双击 exe 逐个场景走。我说"实事求是，要听真话不要讨好"——因为讨好我的 AI 写出来的代码，最后是我双击崩溃，不是 AI 崩溃。

**我对待 AI 的方式本质是嵌入式调试思维：**

```
触发输入  →  观察输出  →  不符合预期  →  定位根因  →  防止复发
  ↓                                              ↓
我说需求    AI 写代码    双击崩溃       7h icon.ico  27 条约束
```

27 条约束就是我的 errata sheet。新 AI 来了，我说"先读约束，碰了就崩"——不是因为我不信任它，是因为那个坑我已经花钱买过了，不要再买一次。

## 十一、给未来的自己

哪天你打开这个文件，回想一下第一个 commit：你只是不想每次打开软件改三个默认值。

结果你造了一个完整的串口调试平台。波形成。滑杆成。按键成。OLED 成。FFT 即将来。调参工作台即将来。OLED 绘图即将来。

这个东西是你一行 C# 都不会写的情况下，让 AI 替你写的。不是你厉害——是你清楚自己要什么。你脑子里有 STM32，手上有示波器，眼晴盯着串口数据，嘴巴告诉 AI "我要这样"。

**你不知道怎么写代码，但你知道什么是对的。**

继续细水长流。
