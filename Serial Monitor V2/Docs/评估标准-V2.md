# Serial Monitor V2 — 评估标准

> **用途**：供新 AI 对话独立对 V2 工程进行多维度打分。
> **使用方式**：先读 `Serial V2 Initial/当前状态.md` 建立全局认知，再按本文档逐维审查代码并打分。

---

## 评估方法论

### 原则
1. **基于代码审查，不依赖实际运行**——评审者无需 STM32 硬件和串口环境
2. **以用户视角为原点**——用户是新手小白，双击 exe 即用；不与 VS2022/命令行/Web 工具对比
3. **承认约束的价值**——WPF 固有限制（142MB 自包含、无裁剪）不应扣分
4. **区分"代码问题"和"架构局限"**——如 WPF 单文件体积是框架固有问题，不应扣代码分
5. **区分"用户决策"和"功能缺失"**——用户明确表示不需要的功能（如多串口、OLED 画图）不扣分
6. **区分"AI 设计选择"和"项目缺陷"**——AI 引导用户采用的方案（如 PNG 图标、面板重建模式）不应扣用户项目分
7. **同一根因不跨维度重复扣分**——一个实现问题（如 `RefreshKeysUI()` 全量重建）应在其根因所属维度扣分，不在其他维度全额重复扣分。跨维度后果可轻微标记（-1）但不应各维独立扣满
8. **个人项目不套企业标准**——无 DI 容器、无 CI/CD 在个人工具项目中是合理选择
9. **"静默处理"不总是缺陷**——串口数据天生含噪音，逐条警告会刷屏。当静默处理是用户明确倾向（如不想弹窗、不想刷日志），扣分应降至最低（-0~-1），仅扣"缺少一次性提示或统计计数"

### 评审流程
1. 读 `Serial V2 Initial/当前状态.md` 前两节（项目概览 + 硬性约束）
2. 通读 `Core/Services/`、`Core/ViewModels/`、`MainWindow.*.cs`、`MainWindow.xaml`
3. 按下方 6 个维度逐项打分
4. 每个维度给出 **得分** + **3 条以内的关键证据**（正面和负面）
5. 加权求和得出总分

### 评分锚点

| 分数 | 标准 |

### 扣分指南
- 无波形交互（缩放/拖拽/游标）：-3（⚠️ 缩放/拖拽已在暂停态可用则仅 -1 游标）
- 无多串口：-2（⚠️ 用户明确不需要则不扣分）
- 无脚本/宏：-1
- OLED 只能显示文本不能画图：-1（⚠️ 用户从未要求画图则不扣分）

## 四、健壮性（权重 15%）

### 评分锚点

| 分数 | 标准 |
|:---:|------|
| 95-100 | 完善的异常处理链、自动恢复、数据完整性保证、全面边界测试 |
| 85-94 | 多层异常分类、资源上限保护、关键路径有 try-catch、崩溃可追踪 |

| <60 | 常见操作崩溃 |

### 审查清单

| 检查项 | 审查位置 |
|--------|---------|
| 串口打开 5 层异常 + 系统日志 | `OpenSerialPort()` |
| 后台线程 Read() try-catch | `SerialPortSession.OnDataReceived()` |
| 发送 Write() try-catch | `SerialPortSession.SendBytes()` |
| 协议路由 try-catch | `MainWindow.OnLineReceived()` |
| SendRaw try-catch | `MainWindow.SendRaw()` |
| 日志行数上限 | `MaxLogLines = 2000` |
| 暂停缓冲上限 | `MaxPausedLines = 2000` |
| 崩溃日志双通道 | `App/App.xaml.cs` |
| 死锁防护 | DataReceived 退订在 Close 之前 |
| 自动重连超时 | 500ms × 10 次 |

### 扣分指南
- 暂停缓冲满时静默丢数据：-1（⚠️ 弹窗打扰比丢数据更差——缓冲满时状态栏已显示计数，用户可感知。扣分仅针对首次满时无一次性提示）
- 协议消息无合法性校验（如 key 的 state 不校验 "down"/"up"）：-2
- HexToBytes 接收侧静默剥离非法字符：-1（⚠️ 串口噪音不可避免，逐条警告会刷屏。发送侧已有实时提醒，接收侧静默是合理的——仅扣"未在状态栏统计被剥离字节数"）
- 无 CRC/数据校验：-1
