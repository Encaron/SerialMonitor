# 串口助手 WPF 压力测试计划

> **目标**：系统性地测试软件在各类极限场景下的表现，验证批量刷新、行号重建、裁剪逻辑等关键路径是否正确。
> **环境**：STM32 + Serial 库（Serial.h/Serial.c）+ 串口助手 WPF

---

## 测试前准备

每个测试用例给出 STM32 端代码片段，直接放入 `while(1)` 或初始化段即可。
建议在测试前先 `git commit` 存档，方便出错回滚。

---

## 一、接收性能

### 1.1 持续极高速率（10ms 间隔）
**目的**：测试批量刷新在极限频率下是否流畅、不卡死、不丢帧

```c
while (1) {
    Serial_Printf(&huart1, "Hello Serial!\r\n");
    HAL_Delay(10);
}
```

**观察点**：
- [ ] 数据是否以 ~80ms 的间隔分批刷新（而非 4 秒 200 行卡一块）
- [ ] 到 2000 行后裁剪是否平滑
- [ ] 软件不卡死，滚动流畅
- [ ] TX/RX 流量计数是否正常增长

---

### 1.2 中高速率（50ms 间隔）
**目的**：日常调试场景，确认流畅度

```c
while (1) {
    Serial_Printf(&huart1, "Hello Serial!\r\n");
    HAL_Delay(50);
}
```

**观察点**：
- [ ] 行号连续递增无跳号
- [ ] 数据看起来接近实时

---

### 1.3 低速间隔（1000ms）
**目的**：确认 debounce→固定间隔改造后，低速场景不受影响

```c
while (1) {
    Serial_Printf(&huart1, "Hello Serial!\r\n");
    HAL_Delay(1000);
}
```

**观察点**：
- [ ] 每行独立出现，不堆积
- [ ] 开头不会有一大堆旧数据冲出来（DiscardInBuffer 生效）

---

### 1.4 零延迟轰炸（极端）
**目的**：测试 50 行紧急刷新阈值是否正常工作，会不会 OOM

```c
while (1) {
    Serial_Printf(&huart1, "Hello Serial!\r\n");
    // 无 HAL_Delay，全速发送
}
```

**观察点**：
- [ ] 软件是否仍然响应（按钮可点击、窗口可拖拽）
- [ ] 接收区是否每 ~50 行刷新一次
- [ ] 不崩溃、不 OOM

**观察点**：

### 2.2 模拟 ADC 数据（含数字变化）
**目的**：模拟电位器 ADC 旋转时数值连续变化

```c

```

**观察点**：
- [ ] 数值变化连续无跳变
- [ ] 滚动时数据不卡顿

---

### 2.3 模拟连续函数（正弦波）
**目的**：类似串口绘图器的输入数据

```c

while (1) {
    float t = HAL_GetTick() / 1000.0f;
    float val = 3.3f * (sin(2 * 3.14159f * 0.5f * t) + 1) / 2;
    Serial_Printf(&huart1, "%.3f\r\n", val);
    HAL_Delay(20);
}
```

**观察点**：
- [ ] 数据频率约 50Hz，是否持续流畅
- [ ] 2000 行后是否平滑裁剪

---

### 2.4 HEX 模式原始数据
**目的**：测试 HEX 模式下的显示性能

```c
uint8_t counter = 0;

```

**观察点**：
- [ ] HEX 模式切换到 "HEX模式" 后数据是否正确显示
- [ ] 每行 32 字节 HEX 显示是否对齐

## 三、双向通信

### 3.1 发送命令 + 接收响应
**目的**：测试收发同时进行时的稳定性

```c
// STM32 端 — 先初始化接收
UART_InitReceive(SERIAL_DEVICE_1, SERIAL_PROTOCOL, SERIAL_TEXT);

while (1) {
    // 检查是否收到 PC 发来的命令
    if (UART_GetRxFlag(SERIAL_DEVICE_1)) {
        char *data = UART_GetRxData(SERIAL_DEVICE_1);
        Serial_Printf(&huart1, "Echo: %s\r\n", data);

        if (strcmp(data, "LEDON") == 0) {
            HAL_GPIO_WritePin(GPIOC, GPIO_PIN_13, GPIO_PIN_RESET);
            Serial_Printf(&huart1, "LED: ON\r\n");
        } else if (strcmp(data, "LEDOFF") == 0) {
            HAL_GPIO_WritePin(GPIOC, GPIO_PIN_13, GPIO_PIN_SET);
            Serial_Printf(&huart1, "LED: OFF\r\n");
        }
    }
    // 同时持续发送心跳
    static uint32_t last_beat = 0;
    if (HAL_GetTick() - last_beat > 500) {
        last_beat = HAL_GetTick();
        Serial_Printf(&huart1, "Heartbeat: %lu\r\n", last_beat);
    }
}
```

**PC 端操作**：
- [ ] 发送 `@LEDON\r\n` → 应收到 `Echo: LEDON` + `LED: ON`
- [ ] 发送 `@LEDOFF\r\n` → 应收到 `Echo: LEDOFF` + `LED: OFF`
- [ ] 心跳持续显示，不受命令收发影响
- [ ] 发送历史是否正确记录

---

### 3.2 定时发送 + 接收同时进行
**目的**：测试 WPF 端定时发送功能与接收是否冲突

PC 端设置定时发送（如 200ms 间隔），STM32 端持续发送数据。

**观察点**：
- [ ] 接收数据正常更新
- [ ] 定时发送正常触发
- [ ] 消息回显是否与接收数据混淆

## 四、UI 交互压力

### 4.1 暂停/恢复（高压下）
**目的**：确认暂停缓冲区在高速数据下不溢出、恢复时行号正确

操作：打开高速发送（10ms 间隔）→ 暂停 5 秒 → 恢复

**观察点**：
- [ ] 暂停期间界面冻结但不卡死
- [ ] 恢复后补回所有积压数据
- [ ] 行号连续不跳
- [ ] 补回提示的条数正确

---

### 4.2 行号开关切换
**目的**：验证行号重建逻辑

操作：收数 → 关行号 → 再收 10 行 → 开行号

**观察点**：
- [ ] 行号与段落数一一对应
- [ ] 无跳号/断号
- [ ] 起始行号 = _lineCount - Blocks.Count + 1（全局连续）

---

### 4.3 明暗主题切换（高压下）
**目的**：RefreshRichTextBoxColors 在大量段落时是否卡

操作：收到 2000 行后 → 点击切换主题

**观察点**：
- [ ] 切换时间 < 500ms
- [ ] 所有已有文字颜色正确更新
- [ ] 不崩溃

---

### 4.4 窗口拖拽 / 面板缩放（高压下）
**目的**：确认 GridSplitter 和布局在数据刷新时不卡

操作：高速接收数据时，拖拽左右面板分割线、拖拽窗口边缘

**观察点**：
- [ ] UI 响应正常
- [ ] 接收区文本渲染正常

---

### 4.5 清空接收区（高压下）
**目的**：确认清空操作正确处理批量队列和行号

操作：高速接收 → 点击"清空接收区"

**观察点**：
- [ ] 接收区和行号面板同时清空
- [ ] 清空后新数据行号从 1 重新开始
- [ ] 批量队列也被清空（不会有延迟数据被写进来）

## 已知注意

- 1.4（零延迟）预期行为：每 ~50 行批量刷新一次（紧急阈值），不会像旧版逐行刷新那样流畅，但**不应卡死**
- 所有测试中如果发现卡死，记下当时的行数、发送间隔、操作步骤
- 如果某个测试通过但体验不佳（如刷新间隔太长），可以回来调整 `_batchFlushTimer` 的间隔（当前 80ms）
