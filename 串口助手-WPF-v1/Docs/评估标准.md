# 串口助手 WPF — 评估标准

> **用途**：供新 AI 对话在没有历史记忆的情况下，独立、可复现地对工程进行多维度打分。
> **使用方式**：先读 `当前状态.md` 建立全局认知，再按本文档逐维审查代码并打分。

---

## 评估方法论

### 原则
1. **基于代码审查，不依赖实际运行**——评审者无需 STM32 硬件和串口环境
2. **以用户视角为原点**——用户是新手小白，双击 exe 即用；不与 VS2022/命令行/Web 工具对比
3. **承认约束的价值**——受限环境下的优秀决策（如 C# 7.3、partial class 拆分）应加分而非扣分
4. **区分"代码问题"和"架构局限"**——如 RichTextBox 性能瓶颈是 WPF 固有问题，不应扣代码分

### 评审流程
1. 读 `串口助手-WPF/当前状态.md` 前两节（项目评估 + 硬性约束）
2. 通读 6 个 .cs 文件 + MainWindow.xaml
3. 按下方 6 个维度逐项打分
4. 每个维度给出 **得分** + **3 条以内的关键证据**（正面和负面）
5. 加权求和得出总分
6. 输出评价文字 + 改进建议

## 一、功能完整性（权重 25%）

### 评分锚点

| 分数 | 标准 |
|:---:|------|
| 95-100 | 专业级串口工具全部功能 + 搜索/绘图/脚本/多串口 |
| 85-94 | 所有常用功能 + 便捷增强（快捷发送/HEX格式化/时间戳/暂停/流控/热插拔） |
| 75-84 | 基本收发 + 常用功能（编码/HEX/文本模式/行号/流量统计/暗色主题） |
| 60-74 | 基本收发可用，功能有缺失 |
| <60 | 无法完成基本串口通信 |

| 功能 | 审查位置 |

| 时间戳（3 格式） | `MainWindow.Log.cs` → `LogReceived` |
| 定时发送 | `MainWindow.xaml.cs` → `chkAutoRepeat_Changed` |

### 扣分指南
- 搜索未实现：-3 ~ -5（AvalonEdit 已就位，一行代码的事）
- 无串口绘图功能：-2 ~ -3（高级功能）
- 无脚本/宏：-1 ~ -2（专业级功能）
- 无多串口：-1 ~ -2

## 二、代码架构（权重 20%）

### 评分锚点

| 分数 | 标准 |
|:---:|------|
| 95-100 | 清晰分层、依赖注入、接口抽象、完整测试、零重复代码 |
| 85-94 | 良好的职责分离、合理的文件拆分、一致的命名、大部分有文档 |
| 75-84 | 基本拆分、少量重复、主要路径有注释 |
| 60-74 | 可工作但大文件堆积、方法过长、命名不一致 |
| <60 | 无法维护的意大利面代码 |

### 审查清单

| 检查项 | 审查位置 |
|--------|---------|
| 文件拆分合理性 | 6 个 partial class 文件的职责边界 |
| 最大文件行数 | `MainWindow.xaml.cs` 行数 |
| XML 注释覆盖率 | 各 .cs 文件中的 `/// <summary>` |
| 设计系统一致性 | `MainWindow.xaml` Window.Resources + `MainWindow.xaml.cs` ThemeMap |
| 常量管理 | `EmergencyFlushThreshold` / `CropMargin` / `SmartScrollLockPixels` / `BatchFlushIntervalMs` / `MaxLogLines` / `MaxPausedLines` |
| 知识库文档 | `当前状态.md` 的完整性和时效性 |
| 架构决策记录 | "关键决策速查表"是否有原因说明 |
| 角色追踪机制 | `_lineRoles` + `LogColorizer` 的同步保证 |

### 关键架构决策评估
审查以下决策的 **原因合理性**：

| 决策 | 评估要点 |
|------|---------|
| partial class 而非 Service 类 | 是否适合"新手小白用户"的维护场景？ |
| AvalonEdit 替代 RichTextBox | 是否显著降低了代码复杂度？ |
| `LogColorizer` + `_lineRoles` 替代 `Paragraph.Tag` | 同步逻辑是否正确？ |
| 去掉批量队列（`_logBatch`） | 是否能正确处理高频数据？ |
| `DataReceived -=` / `+=` 生命周期 | 是否解决了死锁？

### 扣分指南
- 主文件 >1000 行：-3 ~ -5
- 零单元测试：-5 ~ -8
- 部分逻辑耦合（如 UI 事件处理混在串口核心文件中）：-2 ~ -4
- 文档过时（与代码不符）：-5（严重）
- C# 版本受限但合理利用：不扣分

## 六、性能优化（权重 10%）

### 评分锚点

| 分数 | 标准 |
|:---:|------|
| 95-100 | 微秒级延迟、虚拟化渲染、惰性加载、内存 <10MB 稳定 |
| 85-94 | 核心路径优化到位、虚拟化渲染、线性 O(n) 操作 |
| 75-84 | 有关键优化（批处理/裁剪）、基本流畅 |
| 60-74 | 可感知的卡顿、无显著优化 |
| <60 | 严重卡顿、内存泄漏 |

### 审查清单

| 检查项 | 审查位置 |
|--------|---------|
| 渲染架构 | AvalonEdit 虚拟化（`ICSharpCode.AvalonEdit`）是否替代了 RichTextBox FlowDocument？ |
| 追加操作复杂度 | `AppendColoredLine` → `editor.Document.Insert()` — 是否为 O(1) 字符串插入？ |
| 裁剪操作复杂度 | 是否为一次 `Document.Remove()` 而非循环 Paragraph 删除？ |
| 是否消除了布局重算 | 不再操作 `rtReceive.Document.Blocks` / `Paragraph` / `FlowDocument`？ |
| 内存管理 | `MaxLogLines = 2000` / `MaxPausedLines = 2000` / `lbSystemLog` ≤50 |
| 线程模型 | DataReceived（线程池）+ Dispatcher.Invoke（UI 线程）是否合理？ |

### 关键对比
阅读代码时注意：**是否存在以下 RichTextBox 时代的操作？** 如果存在则应扣分。

| 操作 | 预期状态 |
|------|---------|
| `rtReceive.Document.Blocks.Add(para)` | ❌ 不应存在（已迁移到 AvalonEdit） |
| `new Paragraph()` | ❌ 不应存在 |
| `rtReceive.Document.Blocks.Remove(...)` | ❌ 不应存在 |
| `FindVisualChild<ScrollViewer>(rtReceive)` | ❌ 不应存在 |
| `_batchFlushTimer` / `_logBatch` / `FlushLogBatch()` | ❌ 不应存在（已简化为直接追加） |
| `icLineNumbers.Items` | ❌ 不应存在（已用 AvalonEdit 原生行号） |
| `svLineNumbers` / `UpdateLineNumberColumnWidth` | ❌ 不应存在 |
| `editor.Document.Insert(editor.Document.TextLength, text)` | ✅ 应该存在 |
| `LogColorizer : DocumentColorizingTransformer` | ✅ 应该存在 |
| `_lineRoles.Add(role)` | ✅ 应该存在 |

### 扣分指南
- 仍在使用 RichTextBox / FlowDocument / Paragraph：-15（架构退化，严重）
- 裁剪用循环 Remove 而非一次 Document.Remove：-5
- 存在 `FindVisualChild`：-3（AvalonEdit 不需要）
- 追加时创建不必要的中间对象：-2
